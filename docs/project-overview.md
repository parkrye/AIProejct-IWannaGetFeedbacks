# AI 다중 페르소나 SNS 피드백 생성기 — 프로젝트 정리

> 로컬에서 무료로 돌아가는, 502개 페르소나가 각자의 말투로 댓글을 다는 "댓글 봇 실험실".
> 2026-03-17 ~ 2026-03-18 동안 집중 개발. 이 문서는 `/logs/` 18건과 각 도메인 README를 기반으로 한 정리본이다.

---

## 1. 개발 목적

- **출발점**: "댓글 봇을 한번 만들어보자."
- **가설**: 단일 모델/단일 프롬프트로 댓글을 찍어내면 지루하고 획일적이다. **여러 페르소나가 각자의 시각으로 반응**하면 실제 커뮤니티 댓글창에 더 가까워질 것이다.
- **제약 조건**: **무료 + 로컬 실행**. API 비용 없이, 내 PC에서 끝낸다.
- **결과물**: SNS 게시글(텍스트+이미지) → 502개 페르소나 중 n명 선발 → 각자의 성격/나이/말투로 댓글 n개 생성.

## 2. 사고 흐름

프로젝트 전반에 걸쳐 몇 차례 관점의 전환이 있었다.

### 2.1 "페르소나는 얼마나 많아야 충분한가?"
- 초기 5개 페르소나(밝은 응원단/차분한 분석가/재치/다정한 이웃/까다로운 비평가)로 출발.
- 실제 댓글창의 다양성에 전혀 못 미침 → **102개**로 확장 → 여전히 스펙트럼이 좁음 → **502개**로 재확장.
- 카테고리를 직업/서브컬처/인물원형/상황/지역-온라인 5축으로 쪼개 병렬 생성.

### 2.2 "대화 데이터가 풍부하니까 RAG를 써볼까?"
- AI Hub에 한국어 대화 데이터셋이 수십만 건 공개되어 있음.
- 프롬프트에 "이렇게 써" 지시만 하는 것보다, **실제 한국어 댓글 말투를 벡터 검색으로 붙여주면 자연스러움이 올라갈 것**이라는 가설.
- SNS 대화 하나만 쓰려 했으나, 스키마가 다른 데이터셋 4종을 통합 → **어댑터 패턴**으로 해결.
- 최종 **305,545건 / 552MB** 벡터 스토어 구축.

### 2.3 "긍정 일변도는 인터넷이 아니다"
- 처음엔 "건전한" 페르소나 위주였으나, 실제 한국 커뮤니티 댓글은 독성/공격성이 상당 비율.
- 독성 비율을 **31% → 55%**로 재조정. 단, "독성+전문성" "독성+지역성" 등 **다차원 조합**으로 평면적이지 않게.

### 2.4 "선택 UX를 어떻게 풀까"
- 502명 중 직접 고르게 하면 피곤함 → **3가지 선택 모드**로 분기.
  - **동적 선택**: 게시글 내용 ↔ 페르소나 관심사 매칭 점수로 자동 선발.
  - **그룹 선택**: 48개 사전 정의 그룹(게임 커뮤니티, 동창회, 지역 주민 등).
  - **직접 선택**: 기존 방식.

### 2.5 "파라미터는 장식이 아니라 실제 동작을 바꿔야"
- 슬라이더(긍정/헛소리/길이/이모지/격식)를 처음엔 프롬프트 문구만 바꾸는 용도로 추가.
- 이후 **실제 추론 파라미터와 연동**: `nonsense` → `temperature` (0.3~1.2), `verbosity` → `maxTokens` (48~256), 관심사 → RAG 검색 필터.

### 4.4 생성 파라미터

| 파라미터 | 0 | 10 | 시스템 연동 |
|---|---|---|---|
| positivity | 부정적 | 긍정적 | 프롬프트 문구 |

- 페르소나의 첫 번째 관심사는 **RAG 검색 필터**로 사용.

### 4.5 선발 공식 (피드백 인원 n, 1~10)

```
m = n * 2              # 랜덤 혼합 임계값
i = ceil(n * 0.6)      # 관심사 매칭 인원
j = n - i              # 무작위 보강 인원
```

| 그룹/풀 크기 | 선발 방식 |
|---|---|
| n명 이하 | 전원 |
| n+1 ~ n+m-1명 | 관심사 매칭 상위 n명 |
| n+m명 이상 | 매칭 i명 + 무작위 j명 |

## 5. 주요 결정과 트레이드오프

개발 중 겪은 실질적 선택과 그 이유.

### 5.1 GPU → CPU 추론 전환
- **증상**: GTX 1660 SUPER(6GB VRAM)에서 14B 모델 + contextSize 4096 → `InsufficientMemoryError`.
- **선택**: `getLlama({ gpu: false })` + contextSize 2048.
- **트레이드오프**: 속도는 느리지만 OOM 없이 누구나 실행 가능. "무료 로컬"이라는 출발점과 일치.

### 5.2 maxTokens 512 → 128
- SNS 댓글 길이에 512는 과잉 → 128로 축소하며 **CPU 추론 속도 ~4배 향상**.

### 5.3 온라인 구어체 1,270만 → 65,000 (카테고리별 5,000 샘플링)
- 전량 임베딩은 메모리/시간 폭증 → 카테고리별 균등 샘플링.
- `--sample-per-category=N`으로 조정 가능하게 설계해 **"하드코딩 금지" 원칙** 준수.

### 5.4 실시간 토큰 스트리밍 → 완료 후 일괄 표시
- 토큰 단위 스트리밍은 화려하지만, 다수 페르소나 병렬 생성 시 UX가 혼란스러움.
- SSE 이벤트를 `progress` / `feedback` / `error` 3종으로 재설계. 완료된 카드만 보여주고 에러 카드는 분리.

### 5.5 부정 페르소나 31% → 55%
- 실제 한국 커뮤니티 댓글 분포에 맞춰 재조정.
- "나쁘지만 착한 면이 있는" 수준이 아니라 **독성이 메인 + 전문성/지역성 등 배경이 복합된** 다차원 설계.

### 5.6 SSE 이벤트 타입 3분화
- 단일 이벤트로 진행/완료/에러를 섞으면 클라이언트 분기 복잡.
- `progress` → 대기 표시, `feedback` → 카드 렌더, `error` → 에러 카드.

## 6. 현재 한계 및 향후 과제 (메모)

- **이미지 분석이 브라우저 전용**: MediaPipe가 프론트에서 돌아 서버는 텍스트 태그만 수신.
- **벡터 스토어가 인메모리 + 단일 파일**: 규모 확장 시 디스크 기반(예: FAISS, sqlite-vss) 전환 필요.
- **CPU 추론 속도**: 14B 모델은 1분에 댓글 몇 개 수준. 7B로 내리거나 GPU 옵션 조건부 복원 고려.
- **페르소나 품질 검증 자동화 없음**: 502개가 실제로 서로 구분되는 말투를 내는지는 수동 확인에 의존.
